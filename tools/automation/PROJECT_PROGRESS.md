## Current Role

`tools/automation` is the local workspace automation package for `/Users/yumei`; it is not yet an independent reusable project or public release unit.

Its job is to keep the workspace safe, traceable, and handoff-ready across Mac, Windows, Mac mini, coco, and VPS-adjacent workflows.

## Safety Boundary

- No commit, push, PR, merge, deploy, sync, launchd mutation, remote mutation, VPS mutation, or secret access unless explicitly approved.
- Runtime state under `runtime/`, generated reports, logs, trace ledgers, and machine state remain local-only unless explicitly reclassified.
- VPS scripts and sync scripts are high risk even when they have `--dry-run`; changing them requires documenting impact and running the smallest safe local verification.
- `PROJECT_PROGRESS.md` is only a compact pointer surface. Detailed evidence belongs in runtime artifacts and trace ledgers.

## Canonical Entry Points

- `scripts/automationctl status`: summarize full-chain, board, dry-run, execute-local, governance, and OpenCode evidence.
- `scripts/automationctl refresh`: refresh local control-plane reports without approvals, runner confirmation, worker sync, deploy, or remote mutation.
- `scripts/automationctl refresh --run-dry-run`: additionally refresh allowed dry-run evidence; still no real execution.
- `scripts/automationctl validate`: run the standard Phase F validation gate.
- `scripts/automationctl validate --full`: also run full workspace automation validation.
- Telegram daily surface should converge to `/ai` for conversational intake and `/center` for approval/status operations. Legacy Phase F/H/OpenCode/task-board commands are internal/advanced.

## Default Low-Risk Auto Loop

Use this bounded flow for new low-risk issues only after readiness says it is safe to continue:

- Analyze control-plane state and trace history first.
- Generate local candidates with `scripts/opencode-bug-discovery.py --dry-run --project tools/automation`.
- Import exactly one low-risk candidate with `scripts/import-opencode-tasks.py --source bug-discovery --proposal-id <id> --max-risk low --apply`; this grants `plan` only.
- Refresh the board, then run `scripts/auto-dry-run-selector.py --include-manual --run --run-manual`.
- Apply `scripts/execute-local-gate.py --task-id <task-id> --apply` only when blockers are empty and evidence is fresh.
- Execute only that single task through `scripts/runner-two-stage-confirm.py preview` and `confirm`.
- Refresh reports, validate, run changed-file review, and write trace.

## Main-Agent / Subagent Mode

- Main agent coordinates, reads summaries, enforces gates, and writes trace.
- Subagents receive compact packets generated by `scripts/multi-agent-dispatch.py` plus role guides in `workspace-guides/multi-agent/`.
- Planner subagents split ambiguous work into bounded dev-control tasks and do not edit code.
- Developer subagents make the smallest source/test/doc changes inside allowed paths only.
- Tester subagents run validation only and do not edit files.
- Reviewer subagents perform read-only changed-file review and report findings first.
- Main agent reads only changed paths, pass/fail, <=200-word digest, and artifact paths from subagents.
- Long runs must use `scripts/long-auto-loop-supervisor.py` and compact reports from `scripts/long-run-digest.py`.

## Boundary Decision

Keep this automation architecture in this package for now instead of splitting it into a separate project.

Reasons:

- It is tightly coupled to `/Users/yumei` paths, Windows path mappings, local launchd, `.omx` cluster state, and trace ledgers.
- Splitting now would add versioning, publication, sync, and sensitivity-classification work before interfaces are stable.
- A future split remains possible after source/runtime/machine-specific boundaries stabilize.
